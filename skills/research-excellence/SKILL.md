# Chronos Lab Impeccable Research Skill

**版本:** v1.0  
**创建时间:** 2026-03-12  
**灵感来源:** [Impeccable by Paul Bakaus](https://github.com/pbakaus/impeccable) (前端设计质量保障)  
**转化领域:** 前端/UI 设计 → 学术研究

> **致谢:** 本技能方法论框架源自 [Impeccable](https://github.com/pbakaus/impeccable) 项目，经授权转化应用于学术研究领域。原项目专注于前端设计质量保障，本技能将其核心思想 (技能 + 命令 + 反模式) 适配到研究场景。

---

## 🎯 为什么需要 Impeccable Research？

### 问题根源

每个 AI 都从相同的通用模板学习。没有指导，就会产生可预测的错误：

| 领域 | 设计领域 (Impeccable 原项目) | 研究领域 (本技能) |
|------|------------------------------|------------------|
| **字体/表达** | Inter 字体、系统默认 | jargon 掩盖空洞、模糊定性 |
| **颜色/可视化** | 紫色渐变、灰字彩色背景 | 缺少图表、数据可视化不足 |
| **布局/结构** | 卡片套卡片、嵌套过度 | 长段落无结构、逻辑链断裂 |
| **动效/流程** | bounce 缓动、 dated 效果 | 缺少动态演示、流程不清晰 |
| **反模式** | 纯黑灰、过度设计 | 不可检验、忽略反例、过度外推 |

### Impeccable Research 对抗这些偏见

- **1 个核心技能** — 研究质量保障 (7 个领域参考)
- **17 个命令** — 审计、评审、精炼、深化
- ** curated 反模式** — 明确告诉 AI 不该做什么
- **质量阈值** — 综合≥0.85 才允许发表

## 🔧 17 个研究命令

### 质量保证命令 (5 个)

| 命令 | 功能 | 使用场景 | PPT 应用 |
|------|------|----------|----------|
| `/audit` | 运行技术质量检查 (逻辑/可检验/原创) | 研究循环完成后 | 质量审计幻灯片 |
| `/critique` | 深度评审：理论深度、原创性、影响力 | 论文/假设生成后 | 评审意见汇总 |
| `/normalize` | 与 Chronos Lab 标准对齐 (术语/格式) | 多 Agent 协作后 | 标准化报告 |
| `/polish` | 最终打磨，准备发表 | arXiv 提交前 | 最终演示稿 |
| `/harden` | 加强边界条件、反例处理、局限性 | 理论完善阶段 | 风险评估页 |

### 精炼优化命令 (4 个)

| 命令 | 功能 | 使用场景 | PPT 应用 |
|------|------|----------|----------|
| `/distill` | 提炼核心，去除冗余 | 长文档→摘要 | 1 页摘要幻灯片 |
| `/clarify` | 改进模糊表述，增加精确度 | 复杂理论解释 | 清晰概念图 |
| `/optimize` | 优化研究效率 (Token/间隔) | 系统性能调优 | 效率报告 |
| `/extract` | 提取可复用组件/模块 | 多项目复用 | 组件库展示 |

### 深化整合命令 (4 个)

| 命令 | 功能 | 使用场景 | PPT 应用 |
|------|------|----------|----------|
| `/deepen` | 增加理论深度，追溯第一性原理 | 表面分析→深度 | 深度分析页 |
| `/connect` | 建立跨领域连接，发现隐藏关联 | 跨学科整合 | 关联网络图 |
| `/synthesize` | 整合多源结果，生成统一叙述 | 多循环整合 | 综合结论页 |
| `/contextualize` | 定位学术脉络，对比现有工作 | 文献综述 | 学术定位图 |

### 实证锚定命令 (4 个)

| 命令 | 功能 | 使用场景 | PPT 应用 |
|------|------|----------|----------|
| `/test` | 生成可检验预测，设计实验方案 | 理论→实证 | 实验设计页 |
| `/challenge` | 主动寻找反例和替代解释 | 自我批判 | 反例处理页 |
| `/quantify` | 增加数学形式化和量化指标 | 定性→定量 | 公式/数据页 |
| `/visualize` | 生成图表、框架图、概念地图 | 可视化表达 | **核心图表生成** |

## ⚠️ 研究反模式 (Anti-Patterns)

### 理论构建反模式 (5 条)

| ❌ 避免 | ✅ 推荐 | PPT 检查项 |
|--------|--------|------------|
| 模糊定性表述 | 数学形式化 | 每页有公式/数据？ |
| 忽略已知反例 | 主动处理 | 有反例讨论页？ |
| 过度外推 | 明确适用范围 | 有边界条件说明？ |
| 循环论证 | 检查逻辑链 | 逻辑链可视化？ |
| 忽视替代理论 | 公平对比 | 有对比分析页？ |

### 实证锚定反模式 (5 条)

| ❌ 避免 | ✅ 推荐 | PPT 检查项 |
|--------|--------|------------|
| 不可检验预测 | 必须可证伪 | 有可检验预测页？ |
| 忽略测量误差 | 量化不确定性 | 有误差棒/置信区间？ |
| cherry-pick 数据 | 全面评估 | 有完整数据集？ |
| 混淆相关与因果 | 谨慎推论 | 有因果机制图？ |
| 忽略基线对照 | 设置合理对照 | 有对照组数据？ |

### 表达呈现反模式 (5 条)

| ❌ 避免 | ✅ 推荐 | PPT 检查项 |
|--------|--------|------------|
| jargon 掩盖空洞 | 清晰优先 | 外行能看懂？ |
| 过度被动语态 | 主动表达 | 语言简洁有力？ |
| 长段落无结构 | 分段 + 小标题 | 每页≤5 行文字？ |
| 忽略可视化 | 图表胜千言 | **每页有图？** |
| 隐藏局限性 | 诚实透明 | 有局限性说明页？ |

### 跨学科整合反模式 (5 条)

| ❌ 避免 | ✅ 推荐 | PPT 检查项 |
|--------|--------|------------|
| 生搬硬套概念 | 深度理解再应用 | 有概念溯源？ |
| 忽略领域特定约束 | 尊重专业 | 有领域专家验证？ |
| 过度简化复杂问题 | 保持复杂性 | 有复杂度说明？ |
| 忽视术语差异 | 精确翻译 | 有术语对照表？ |
| claim 统一但拼凑 | 真正整合 | 有统一框架图？ |

## 🔄 集成到研究循环

### Deep-Cycle 标准流程 (6 阶段)

```
阶段 1: 上下文加载 (~900k tokens)
阶段 2: 深度分析 (跨领域连接)
阶段 3: 综合推演 (假设/问题生成)
阶段 4: 知识固化 (卡片/实验方案)
阶段 5: /audit + /critique ← 质量保证
阶段 6: /polish + /harden ← 最终打磨
```

### 论文写作流程

```
草稿 v0.5 → /critique → 修订 v1.0
修订 v1.0 → /harden → 修订 v1.5
修订 v1.5 → /polish → 提交 arXiv
```

### PPT/报告生成流程

```
研究内容 → /distill → 核心要点
核心要点 → /visualize → 图表生成
图表 + 要点 → /polish → 最终演示稿
```

## 🧠 心智模型

### Impeccable Research 的核心原则

1. **深度优先于速度** — 但不牺牲效率
2. **实证优先于思辨** — 但不扼杀想象力
3. **清晰优先于复杂** — 但不简化本质
4. **诚实优先于自信** — 量化不确定性
5. **整合优先于拼凑** — 真正统一框架

### PPT/报告设计原则 (转化自 Impeccable)

1. **1 页 1 核心** — 每页幻灯片只传达 1 个核心信息
2. **图表胜千言** — 优先使用可视化，文字≤5 行/页
3. **对比产生意义** — 前后对比、理论对比、数据对比
4. **留白即呼吸** — 避免信息过载，给观众思考空间
5. **一致性即专业** — 字体/颜色/布局保持统一

## 📊 PPT/报告模板 (直接套用)

### 研究汇报标准结构

```
1. 标题页 (研究问题 + 核心洞见 1 句话)
2. 动机页 (为什么重要？现有理论不足)
3. 框架总览图 (Figure 1: ITLCT 统一框架)
4. 核心方程页 (3-5 个关键公式)
5. 实验验证页 (路线图 + 时间轴)
6. 质量审计页 (6 维度雷达图)
7. 反例与挑战页 (主动处理质疑)
8. 结论页 (核心贡献 + 下一步)
```
