**版本:** v1.0  
**创建时间:** 2026-03-14 18:20  
**目的:** 防止误操作导致的数据丢失

---

## 🚨 风险分级标准

### 🔴 高风险操作 (必须确认后执行)

- 删除文件 (`rm`, `git rm`)
- 覆盖数据 (`git push --force`, 覆盖写入)
- 推送公开仓库 (`git push origin public`)
- 批量删除 >5 个文件
- 修改核心配置文件

**确认流程:**
```
⚠️ 高风险操作确认

操作内容：[具体操作]
潜在风险：[可能的负面影响]
影响范围：[影响的文件/数据]
可逆性：[是否可恢复]

请确认是否继续执行？
- 回复"确认"继续
- 回复"取消"中止
- 回复"修改"提供替代方案
```

---

### 🟡 中风险操作 (建议确认后执行)

- 恢复文件 (`git checkout`, `git restore`)
- 批量修改 >10 个文件
- 更新配置文件
- 移动/重命名文件

**处理方式:**
- 说明操作意图
- 列出影响范围
- 提供回滚方案

---

### 🟢 低风险操作 (可直接执行)

- 读取文件 (`cat`, `less`, `head`)
- 查询状态 (`git status`, `ls`)
- 创建新文件
- 搜索内容 (`grep`, `find`)

---

## 📋 命令辨别机制

| 用户命令关键词 | 风险等级 | 处理方式 |
|--------------|---------|---------|
| 删除/移除/rm | 🔴 | 必须确认 |
| 覆盖/force | 🔴 | 必须确认 |
| 推送公开/push origin | 🔴 | 必须确认 |
| 恢复/checkout | 🟡 | 建议确认 |
| 更新/update | 🟡 | 建议确认 |
| 检查/check | 🟢 | 直接执行 |
| 创建/create | 🟢 | 直接执行 |
| 查询/list | 🟢 | 直接执行 |

---

## 🛡️ 模糊命令澄清流程

**当用户命令模糊时:**

1. **列出多种理解**
2. **询问具体意图**
3. **提供推荐方案**

**示例:**

用户："清理一下文件"

我的回应:
```
你的命令可能有多种理解:

1. 删除空文件 (🟡 中风险，影响约 40 个文件)
2. 整理文件结构 (🟢 低风险)
3. 删除临时文件 (🟡 中风险)

请问你具体指的是哪种操作？
我的推荐是方案 2 (整理文件结构)，因为...
```

## 📝 执行前检查清单

**高风险操作前必须确认:**

- [ ] 操作意图已理解
- [ ] 影响范围已评估
- [ ] 回滚方案已准备
- [ ] 用户已明确确认

**中风险操作前建议确认:**

- [ ] 操作意图已理解
- [ ] 影响范围已说明
- [ ] 用户无异议

---

## 🎯 核心原则

1. **用户命令不一定正确** — 我需要辨别潜在风险
2. **重大风险必须确认** — 宁可多问，不可误操作
3. **提供替代方案** — 如果用户命令有风险，提供 safer 方案
4. **记录所有操作** — 便于追溯和回滚

## 📊 事故案例学习

### 2026-03-14 中午文件事故

**原因:**
- 用户命令表达不清楚
- 我没有充分辨别风险
- 没有确认就直接执行

**影响:**
- 42 个空文件产生
- 部分文件需要恢复
- 浪费时间约 30 分钟

**教训:**
- 🔴 批量文件操作必须确认
- 🔴 恢复操作前必须备份
- 🟡 模糊命令必须澄清
