## 🎯 关键发现

### 15 分钟间隔的问题

**理论并发：**
```
00:00 → 研究循环启动 (900s timeout)
00:15 → 研究循环 + 文献搜索 同时启动
00:30 → 研究循环 + 文献搜索 + 研究日志
```

**实际并发：** 3-4 个 subagent 同时运行

**模型服务上限：** ~2 个并发 subagent

**结果：**
- HTTP 500 错误（模型服务不可用）
- 超时错误（12 分 35 秒超时）
- 任务堆积（队列阻塞）

---

### 30 分钟间隔的优势

**理论并发：**
```
00:00 → 研究循环启动
00:15 → 文献搜索启动（研究循环已完成）
00:30 → 研究循环 + 研究日志
```

**实际并发：** 1-2 个 subagent

**结果：**
- 错误率 < 5%
- 无超时
- 任务不堆积
- 系统稳定

## 💡 优化原则

### 原则 1: 可持续 > 爆发式

```
❌ 错误做法：15 分钟间隔 → 20% 错误率 → 实际 60 次/日
✅ 正确做法：30 分钟间隔 → 5% 错误率 → 实际 46 次/日
```

### 原则 2: 质量 > 数量

```
❌ 错误做法：96 次浅层循环 → 3 星质量
✅ 正确做法：48 次深度循环 → 4-5 星质量
```

### 原则 3: 稳定性 > 极限速度

```
❌ 错误做法：追求理论极限 → 系统崩溃
✅ 正确做法：留有余量 → 长期稳定运行
```

## 🎛️ 最佳实践

### 间隔设置公式

```python
optimal_interval = max(
    avg_execution_time * 2,  # 2 倍执行时间
    min_interval,            # 最小间隔（15 分钟）
    gateway_capacity / concurrent_tasks  # Gateway 容量
)

# 当前最优：30 分钟
# - 平均执行时间：9 分钟
# - 2 倍缓冲：18 分钟
# - 取整：30 分钟
```

### 并发控制

```python
max_concurrent_subagents = 2

if current_subagents >= max_concurrent:
    delay_new_task(5 分钟)
```

### 错误率监控

```python
if error_rate > 0.1:  # > 10%
    interval *= 1.5  # 间隔 ×1.5
elif error_rate > 0.2:  # > 20%
    interval *= 2.0  # 间隔 ×2.0
```

## 🧬 进化记录

**这次教训让我们学到：**

1. **理论极限 ≠ 实际可行** — 服务器有物理上限
2. **并发不是越多越好** — 2 个 subagent 是甜点
3. **缓冲时间很重要** — 2 倍执行时间是安全线
4. **错误率是关键指标** — > 10% 必须降速
5. **可持续性优先** — 慢一点，稳一点，走得更远
